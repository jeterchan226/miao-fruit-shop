# 設計：店內平板後台「新訂單即時提醒」

日期：2026-06-26

## 背景與目標

下單成立後，後端已 push 訂單明細給**客戶**（`line_service.send_order_created`）。但店家端無法即時得知有人下單：LINE 官方帳號的「聊天」列表排序與通知由使用者主動傳入的訊息驅動，OA 主動 push 出去的訊息不會頂上列表、也不會通知小編；加上訂單走 LIFF 而非聊天訊息，客戶若沒傳訊息給 OA，店家就不會自動感知到訂單。

評估過幾條通知管道（LINE 群組 push、Email、出單印表機）後，決定先做**店內一台平板固定開著訂單後台**的即時提醒：有新訂單就響鈴 + 視覺提示。理由：零硬體、零月費、通知可靠，最快上線；量做大後再評估出單印表機。

目標：平板開著訂單後台時，新訂單即時以**聲音 + 畫面提示**通知店家，並把未處理訂單導引到眼前。

## 決策

- **觸發範圍**：僅「後台開著（前景、螢幕亮）」時負責提醒。平板鎖屏 / 瀏覽器切到背景的通知**不在範圍內**（那需 Web Push + Service Worker + VAPID + 訂閱存 DB + 後端推播，工程明顯更重，YAGNI）。
- **機制**：純前端輪詢既有 API，**後端零改動**。前端每 ~20 秒輪詢 `GET /api/admin/orders?page=1&page_size=10`（既有列表端點，本來就 `created_at desc` 新到舊），用回傳的最新訂單清單同時驅動「未讀數」與通知下拉內容。
- **視覺採 header 通知中心**：在後台 header 放一個鈴鐺 icon + 未讀數徽章（不是 toast / 橫幅）。一筆「未讀通知」= 一筆新訂單。點鈴鐺展開未讀新訂單清單，點清單中的訂單可直接看該筆訂單。
- **未讀狀態以 `localStorage` 標記持久化**：記住「已讀到的最新訂單時間」，比它新的訂單即為未讀。可跨重新整理 / 跨開關後台保留（離線期間進來的單，下次開後台仍顯示為未讀）。
- **另加「查看前台」按鈕**：header 上一顆按鈕，開新分頁連到前台 `/`。
- **通知範圍 = 新訂單**：依使用者明確指示，通知只代表「新訂單」。附圖中的「付款已確認」列為示意，v1 不納入（付款確認是管理者自己改的狀態，不需通知自己）。
- **不做**：Web Push / Service Worker / PWA 安裝、背景通知、LINE 群組推播、出單印表機。皆延後或不採用。

僅前端（`frontend/src/AdminApp.jsx` 及相關）。後端、資料庫、既有客戶通知流程皆不變。

## 變更點（前端）

### 1. 輪詢與未讀計算

在 AdminApp 已登入的外層（不限於訂單分頁）加入輪詢：

- 以 `setInterval` 每 ~20 秒呼叫既有 `listAdminOrders`（`GET /api/admin/orders?page=1&page_size=10`，`created_at desc`），取最近訂單清單供通知中心使用。
- 以 `localStorage` 保存 `admin_notif_last_read_at`（已讀到的最新訂單 `created_at`，ISO 字串）。
- **未讀** = 清單中 `created_at` 比 `last_read_at` 新的訂單；**未讀數** = 其數量，顯示於鈴鐺徽章。
- 首次使用（無 `last_read_at`）→ 以目前最新訂單時間初始化標記，避免把歷史訂單全部當未讀。
- 與上一輪輪詢相比若出現「更新的訂單」（本次 session 真的有新單進來）→ 觸發**聲音**提醒（見 2）。純重新整理 / 首次載入不發聲。
- 元件卸載 / 登出時清除 interval。

### 2. Header 通知中心（鈴鐺 + 未讀清單）

於後台 header 新增鈴鐺 icon：

- **徽章**：顯示未讀數（>0 時才顯示紅點數字；如附圖右下角「3」）。
- **點鈴鐺展開下拉面板**，標題「未讀通知」，列出最近新訂單，每列：`新訂單 {order_no}`、客戶姓名・金額（`NT$ {total}`）・相對時間（如「8 分鐘前」）。未讀列以底色highlight（如附圖綠底），已讀列一般樣式。
- **點清單中的某筆訂單** → 開啟該筆既有的訂單明細 modal（重用 OrderDetail），並將未讀標記推進到該筆（含更舊者視為已讀）。
- 提供「全部標示已讀」動作（把 `last_read_at` 設為最新訂單時間，徽章歸零）。
- 相對時間以輕量自製 helper 計算（秒 / 分 / 小時 / 天前），不引入新套件。

### 3. 聲音提醒

- 有新訂單進來時播放短提示音。
- 瀏覽器自動播放政策會擋未經使用者手勢的音訊。解法：在登入後第一個使用者手勢（登入按鈕點擊，或一個明確的「啟用提醒音 🔔」開關）時先解鎖（預載並以靜音 `play()` 解鎖 `Audio` 物件）。
- 提供開關讓店家可開 / 關提示音，狀態存 `localStorage`。

### 4. 「查看前台」按鈕

- 於後台 header 新增「查看前台」按鈕（附圖含眼睛 icon）。
- 點擊在**新分頁**開啟前台 `/`（`window.open('/', '_blank')` 或 `<a href="/" target="_blank" rel="noopener">`）。前台與後台同屬此前端部署，前台路由為 `/`。

### 5. 防螢幕睡眠（Wake Lock）

- 後台處於前景時，呼叫 `navigator.wakeLock.request('screen')` 讓平板螢幕不自動睡。
- 監聽 `visibilitychange`，回到前景時重新取得 wake lock（鎖會在切背景時自動釋放）。
- 不支援 Wake Lock API 的裝置 → try/catch 略過，不影響其他功能（graceful degradation）。

### 6. 放置位置

輪詢、通知中心狀態、聲音、Wake Lock 邏輯放在 AdminApp 已驗證後的外層元件 / hook，使其**不論當下在訂單或商品分頁都會運作**；鈴鐺與「查看前台」按鈕渲染於後台 header。

## 測試（前端）

1. **未讀計算**：輪詢回傳的最近訂單中，比 `last_read_at` 新的筆數正確反映為未讀數 / 徽章。
2. **偵測新訂單發聲**：出現比上一輪更新的訂單 → 呼叫播放提示音；清單不變 → 不發聲。
3. **首次不誤報**：首次（無 `last_read_at`）以最新訂單初始化標記，徽章為 0、不發聲。
4. **標示已讀**：點某筆訂單 / 「全部標示已讀」→ `last_read_at` 推進、徽章歸零並持久化於 `localStorage`。
5. **音訊解鎖**：未解鎖前不嘗試播放；解鎖後新訂單會呼叫播放。
6. **提示音開關**：關閉時不發聲、仍更新未讀徽章；開關狀態持久化於 `localStorage`。
7. **相對時間 helper**：秒 / 分 / 小時 / 天前格式正確。

（使用假計時器 / mock fetch 進行。）

## 不變更項目

- 後端、資料庫 schema、訂單建立的金額 / 庫存 / 既有客戶通知流程不動。
- 不新增 Web Push / Service Worker / PWA / 背景通知。
- 不新增 LINE 群組推播或出單印表機整合（日後量大再評估）。
